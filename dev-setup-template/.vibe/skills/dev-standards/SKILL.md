# Skill: Development Standards

## Java Spring Boot

### Pflicht-Pattern
```java
// Service immer via Interface
public interface UserService {
    UserDTO findById(Long id);
}

@Service
@RequiredArgsConstructor
public class UserServiceImpl implements UserService {
    private final UserRepository userRepository;

    @Override
    public UserDTO findById(Long id) {
        return userRepository.findById(id)
            .map(UserMapper::toDTO)
            .orElseThrow(() -> new EntityNotFoundException("User not found: " + id));
    }
}
```

### Exception Handling
```java
@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(EntityNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(EntityNotFoundException ex) {
        return ResponseEntity.status(404).body(new ErrorResponse(ex.getMessage()));
    }
}
```

### API-Konventionen
- GET `/api/v1/resources` – Liste (200)
- GET `/api/v1/resources/{id}` – Einzelnes Element (200)
- POST `/api/v1/resources` – Erstellen (201 Created)
- PUT `/api/v1/resources/{id}` – Vollstaendiges Update (200)
- PATCH `/api/v1/resources/{id}` – Partial Update (200)
- DELETE `/api/v1/resources/{id}` – Loeschen (204 No Content)

### Sicherheit
- Alle Endpoints mit `@PreAuthorize` absichern
- Input-Validierung mit `@Valid` + Bean Validation Annotations
- Keine hardcodierten Credentials – immer `@Value` oder Secrets Manager
- Kein `System.out.println` – nur `log.info/warn/error` (SLF4J)

### Verbote
- Kein `@Autowired` auf Feldern (Constructor Injection via Lombok)
- Keine Logik in Entities oder DTOs
- Kein `Optional.get()` ohne `.isPresent()` – immer `.orElseThrow()`
- Keine Magic Numbers – benannte Konstanten verwenden

## Vue.js 3 Composition API

### Komponenten-Template
```vue
<script setup lang="ts">
// 1. Imports
import { ref, computed, onMounted } from 'vue'
import { useUserStore } from '@/stores/user'

// 2. Props & Emits
const props = defineProps<{ userId: number }>()
const emit = defineEmits<{ updated: [user: User] }>()

// 3. Store
const userStore = useUserStore()

// 4. State
const isLoading = ref(false)

// 5. Computed
const user = computed(() => userStore.getById(props.userId))

// 6. Methods
async function loadUser() {
  isLoading.value = true
  await userStore.fetch(props.userId)
  isLoading.value = false
}

// 7. Lifecycle
onMounted(loadUser)
</script>
```

### Verbote
- Kein `this` in Composition API
- Kein direkter API-Call in Komponenten (immer Pinia Store/Composable)
- Keine Options API in neuem Code
- Kein `v-html` mit User-Daten (XSS-Risiko)
- Keine globalen CSS-Overrides

### Testing (Vitest)
Bei jeder neuen/geaenderten Komponente:
- Render-Test: Komponente rendert ohne Fehler
- Props-Test: Alle Pflicht-Props werden korrekt gerendert
- Emit-Test: Events werden bei Interaktion ausgeloest
- E2E-Flows (User-Journey) → Test Agent delegieren

## Oracle Database

### Naming
```sql
-- Tabellen: UPPER_CASE, Plural
CREATE TABLE CUSTOMER_ORDERS (
    ORDER_ID    NUMBER(19) GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    CUSTOMER_ID NUMBER(19) NOT NULL,
    CREATED_AT  TIMESTAMP DEFAULT SYSTIMESTAMP NOT NULL,
    CONSTRAINT FK_ORDER_CUSTOMER FOREIGN KEY (CUSTOMER_ID) REFERENCES CUSTOMERS(CUSTOMER_ID)
);
```

## PostgreSQL

### Naming
```sql
-- snake_case ueberall
CREATE TABLE customer_orders (
    order_id        BIGSERIAL PRIMARY KEY,
    customer_id     BIGINT NOT NULL REFERENCES customers(customer_id),
    created_at      TIMESTAMPTZ DEFAULT NOW() NOT NULL
);
```

### Flyway Migration Naming
```
V001__create_customers_table.sql
V002__add_orders_table.sql
V003__add_index_orders_customer_id.sql
```

### Verbote
- Kein `SELECT *` in Produktion
- Kein DDL ohne Flyway-Migration (niemals manuelles ALTER TABLE)
- Keine Stored Procedures fuer Business-Logik
- Kein `DROP` ohne explizite Bestaetigung
- Keine Migration ohne Index auf Foreign Keys

## Sicherheits-Checkliste (vor jedem Commit)

- [ ] Keine Credentials/Tokens im Code
- [ ] SQL-Queries nutzen PreparedStatements / JPA (keine String-Konkatenation)
- [ ] User-Input wird validiert (@Valid, @NotNull, etc.)
- [ ] API-Endpoints sind mit Rollen abgesichert (@PreAuthorize)
- [ ] Keine sensiblen Daten in Logs
